# 系统架构设计

## 1. 架构目标

MVP 阶段目标是尽快跑通业务闭环，因此优先采用单体架构，而不是微服务架构。设计重点是业务状态、人工审核和权限隔离，不是服务拆分。

## 2. 客户端组成

### 微信小程序
统一承载：
- 用户端
- 陪玩端
- 轻量管理员操作

### Web 后台
承载：
- 俱乐部管理员后台
- 平台运营后台

## 3. 后端架构

推荐技术栈：
- `NestJS`
- `Prisma`
- `MySQL 8`
- `Redis`
- `COS/OSS`

采用单体应用 + 模块化分层：
- `auth`
- `user`
- `club`
- `player`
- `order`
- `report`
- `points`
- `withdraw`
- `reward`
- `admin`
- `platform`
- `notification`
- `audit`

## 4. 分层原则

每个模块建议拆为：
- `controller`：接口和参数校验
- `service`：业务规则和事务
- `repository`：数据库访问封装

关键业务动作必须在服务层完成状态校验和事务控制。

## 5. 权限设计

权限分两层：
- 角色权限：用户、陪玩、俱乐部管理员、平台管理员、超管
- 数据范围：仅能访问自己或所属俱乐部的数据

后端必须根据登录用户实时校验权限，不能信任前端传入的角色信息。

## 6. 异步与缓存

Redis 用于：
- Token/会话辅助
- 热点数据缓存
- 抢单互斥控制
- 简单消息待处理状态

MVP 阶段不强制引入独立消息队列，定时任务和轻异步可先在单体内实现。

## 7. 部署建议

基础部署形态：
- `nginx`
- `api`
- `mysql`
- `redis`
- `object storage`

环境建议：
- `dev`
- `staging`
- `prod`

## 8. 核心非功能要求

- 关键写操作必须可审计
- 关键状态流转必须幂等
- 管理后台必须做数据范围隔离
- 数据库每日备份
- 关键配置修改必须记录操作日志
