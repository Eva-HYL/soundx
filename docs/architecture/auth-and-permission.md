# 鉴权与权限模型

## 1. 设计目标

权限设计必须解决两个问题：
- 谁能做什么
- 谁能看哪些数据

因此本项目权限采用“双层模型”：
- 角色权限控制动作
- 数据范围控制可见性

## 2. 鉴权方式

采用 JWT Bearer Token：

```text
Authorization: Bearer <access_token>
```

Token 只保留最基础信息：
- `sub`：用户 ID
- `iat`
- `exp`

不建议把完整角色与俱乐部范围直接固化进 token，避免角色变更后权限滞后。

## 3. 角色定义

### 用户 `user`
能力：
- 查看自己的资料
- 查看自己所属俱乐部
- 下单
- 查看自己的订单
- 取消符合条件的订单
- 发起打赏

### 陪玩 `player`
能力：
- 维护陪玩资料
- 修改工作状态
- 查看自己可接订单
- 抢单/接单/拒单
- 提交报备
- 查看积分和流水
- 发起提现

### 俱乐部管理员 `club_admin`
能力：
- 查看本俱乐部订单
- 确认订单付款
- 派单
- 审核报备
- 审核提现
- 确认提现转账
- 管理陪玩和俱乐部配置
- 查看本俱乐部统计

### 平台管理员 `platform_admin`
能力：
- 查看平台级数据
- 审核俱乐部
- 配置套餐和功能开关
- 处理平台级异常

### 超级管理员 `super_admin`
能力：
- 拥有系统级配置权限
- 可管理平台管理员权限
- 处理高风险操作

## 4. 数据范围模型

### 用户范围
- 只能访问自己的资料、订单、打赏记录

### 陪玩范围
- 只能访问自己的陪玩资料、接单记录、积分、提现记录
- 只能对自己所属俱乐部的订单执行接单动作

### 俱乐部管理员范围
- 只能访问自己被授权俱乐部的数据
- 不能查看其他俱乐部订单、提现、报备和陪玩数据

### 平台管理员范围
- 默认可查看全平台数据
- 但高风险操作仍应受更细粒度权限约束

## 5. 建模建议

### 5.1 用户角色表 `user_role`
用于表达用户拥有哪些角色，以及角色是否绑定俱乐部范围。

关键字段：
- `user_id`
- `role`
- `club_id`
- `status`

### 5.2 俱乐部后台角色表 `club_staff`
用于表达俱乐部内部后台人员职责，例如：
- `owner`
- `admin`
- `finance`
- `service`

## 6. 权限校验流程

每次请求建议按以下顺序校验：

1. 校验 token 是否有效
2. 获取当前用户基础信息
3. 查询用户角色
4. 判断是否具备目标动作权限
5. 判断是否具备目标数据范围权限
6. 执行业务逻辑

## 7. 典型权限示例

### 用户取消订单
通过条件：
- 请求人是订单创建者
- 订单状态仍允许取消

### 陪玩抢单
通过条件：
- 请求人具备 `player` 角色
- 请求人属于该订单俱乐部
- 请求人当前工作状态可接单

### 管理员确认付款
通过条件：
- 请求人具备对应俱乐部的 `club_admin` 或等效后台角色
- 订单属于该俱乐部
- 订单当前处于 `pending_payment`

### 平台管理员审核俱乐部
通过条件：
- 请求人具备 `platform_admin` 或 `super_admin`

## 8. 实现建议

### 后端
- 使用 NestJS Guard 处理登录校验
- 使用 Decorator + Guard 处理角色校验
- 使用 Service 层做数据范围校验
- 不信任前端传递的角色和俱乐部信息

### 前端
- 前端只做页面级显示控制，不作为安全边界
- 真正权限判断必须在后端完成

## 9. 审计要求

以下权限相关动作必须记录日志：
- 授予或移除管理员角色
- 修改俱乐部后台人员角色
- 修改功能开关
- 进行系统级高风险操作
