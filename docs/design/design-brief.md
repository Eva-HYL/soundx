# SoundX UI 设计 Brief（交付给设计师 / 设计 AI）

> 本文件是 UI 设计工作的输入。可直接复制粘贴给设计师、v0、Figma AI、Galileo 等工具。如果用 AI 工具，建议按"第 10 节：分阶段 Prompt"分批投喂，避免一次生成太多信息失焦。

---

## 1. 项目一句话

SoundX 是一个面向游戏陪玩俱乐部的 SaaS 平台：**俱乐部管理员**通过 Web 后台管理订单/陪玩/财务，**用户**和**陪玩**通过微信小程序下单、接单、服务、结算。平台本身**不接入支付**，所有付款与提现由管理员**人工确认**线下转账。

---

## 2. 平台与端

| 端 | 技术 | 设备基线 | 默认尺寸 | 组件库 |
|---|---|---|---|---|
| **Web 后台** (admin) | React + Ant Design 5 | 桌面浏览器 | 1440 × 900，最小 1280 | Ant Design 5 默认主题（蓝色 `#1677ff`） |
| **微信小程序** (miniapp) | Taro + NutUI-React-Taro | 手机 | 375 × 812（iPhone X 基线） | NutUI 默认主题 |

**重要**：不做自定义组件库，尽量使用 Ant Design / NutUI 原生组件；颜色不自定义，用官方默认主题。设计风格走**简洁干净、表单密度高、业务一目了然**的 B 端 SaaS + 轻量 C 端消费级路线，不追求炫酷。

---

## 3. 用户角色

| 角色 | 端 | 画像 |
|---|---|---|
| **下单用户** | 小程序 | 18-35 岁游戏玩家，主要诉求是便捷找陪玩、明确价格、线下付款后知道自己订单到哪一步 |
| **陪玩** | 小程序 | 18-30 岁在校生/兼职者，诉求是抢到合适单、清晰看到今日收入、顺畅提现 |
| **俱乐部管理员** | Web 后台 | 25-40 岁俱乐部老板/运营，诉求是**快速处理付款确认和报备审核**（高频）、查看陪玩业绩、处理提现 |
| **平台管理员** | Web 后台 | 平台运营，诉求是审批俱乐部、配置套餐、看全局数据 |

**设计优先级**：管理员的"付款确认"和"报备审核"列表是最高频动作，设计上应让这两个页面操作成本最低（批量、一键、键盘快捷键等）。

## 5. 关键流程（必须设计清楚的交互）

设计师必须把下面这些流程走通，每个流程至少给出"关键 3-5 屏"：

### 5.1 下单 → 付款 → 派单（用户 + 管理员协同）

1. **用户**：浏览陪玩 → 选服务/时长 → 提交订单 → 显示订单号和线下转账说明（含俱乐部收款二维码/账户）
2. **用户**：转账后上传付款凭证（图片/PDF，≤5MB）
3. **管理员**：在"待确认付款"列表看到新订单，点进去查看凭证 → 填写"确认到账金额" → 确认
4. 订单状态进入"待派发"。指定单自动派给指定陪玩；普通单进入订单池等陪玩抢

设计要点：用户侧要清晰显示"等待管理员确认付款"的中间态；管理员侧凭证要大图预览，快捷键 `y` 确认、`n` 驳回。

### 5.2 接单 → 服务 → 报备 → 审核 → 结算

1. **陪玩**：订单池抢单 / 被指派接受 → 订单进入 `accepted`
2. **陪玩**：开始服务 → 结束服务 → 填写报备（起止时间、时长、内容、附件）
3. **管理员**：在"待审核"看到报备 → 核对内容 → 输入应发积分（默认=订单金额）→ 通过 / 驳回
4. 通过后订单 `completed`，陪玩积分到账；驳回后陪玩可重新提交

设计要点：陪玩填报备要有计时器辅助；管理员审核要能并排看订单 + 报备内容。

### 5.3 提现（陪玩 → 管理员 → 完成）

1. **陪玩**：查看可提现积分 → 发起提现（金额=积分=元，1:1）→ 填微信号/姓名 → 提交，积分冻结
2. **管理员**：审核提现（驳回则解冻）→ 通过 → 线下转账 → 在系统"确认转账"（上传凭证） → 扣减冻结积分，提现完成
3. **陪玩**：看到状态流转 `待审核 → 待转账 → 已转账 → 已完成`

### 5.4 打赏（可选，MVP）

用户在订单详情/完成页发起打赏 → 上传付款凭证 → 管理员确认到账 → 打赏金额作为积分发给陪玩。

## 6. 状态机可视化需求

以下状态需要在列表 / 详情页用**统一的 tag 颜色**区分（设计师定义一套语义色映射，前后端都遵守）：

**订单状态（10 态）**：
`pending_payment` / `paid_pending_dispatch` / `pending_accept` / `accepted` / `in_service` / `pending_report` / `pending_report_audit` / `completed` / `cancelled` / `closed`

**提现状态（6 态）**：
`pending_review` / `approved_pending_transfer` / `transferred_pending_confirm` / `completed` / `rejected` / `cancelled`

**报备状态（4 态）**：`draft` / `submitted` / `approved` / `rejected`

**打赏状态（5 态）**：`pending_payment` / `paid_pending_confirm` / `confirmed` / `rejected` / `cancelled`

**陪玩工作状态**：`offline` / `online` / `busy` / `resting`

建议颜色语义：
- 待处理（需要我动作）→ **橙色/警告色**
- 进行中 → **蓝色/主色**
- 已完成 → **绿色**
- 已取消 / 驳回 → **灰色 / 红色**

## 7. 视觉与内容规范

### 7.1 颜色

- **后台**：Ant Design 默认 Primary `#1677ff`，其它语义色全部走 AntD 默认
- **小程序**：NutUI 默认主题
- **禁止**：自定义品牌色、渐变色、华丽配色

### 7.2 字体

- 后台：系统默认无衬线，正文 14px，表格 13px
- 小程序：NutUI 默认

### 7.3 文案

- **中文简体**
- B 端后台措辞正式（"确认付款""审核通过"）
- C 端小程序措辞轻快（"等等老板确认哟""战绩已提交"）
- 金额统一格式 `¥80.00`（两位小数），不省略
- 积分统一格式 `80 分`

### 7.4 空状态 / 错误态

每个列表页都需要给空状态插图 + 引导文案；错误页（404 / 500 / 网络错）需统一样式。

---

## 8. 必须体现的业务约束

1. **全部金额两位小数**，`¥80.00` 不要写 `¥80`
2. **1 元 = 1 积分**，UI 文案要能让用户直观理解
3. **付款全部线下**，页面不能出现"立即支付"按钮，只有"我已转账，上传凭证"
4. **文件上传限制**需要在 UI 上说明（凭证 ≤5MB jpg/png/pdf；语音 ≤10MB mp3/aac/m4a；头像 ≤2MB jpg/png/webp）
5. **多俱乐部切换**：用户可能同时属于多个俱乐部（用户+陪玩角色也可能横跨），顶部要有"当前俱乐部"切换器
6. **权限分层**：部分菜单只有特定角色可见（超级管理员 / 平台 / 俱乐部管理员 / 财务 / 客服）

## 10. 分阶段 Prompt（给 AI 设计工具用）

如果交给 v0 / Figma AI / Galileo 等 AI 设计工具，**不要一次投喂整份 Brief**。按下面顺序分批。每批把 1-3 节 + 对应页面需求打包投喂。

### Prompt A：后台付款确认页（最高优先级）

```
请设计一个 SaaS 后台的"待确认付款"页面。技术栈 React + Ant Design 5 默认主题（Primary #1677ff）。

业务背景：游戏陪玩俱乐部的管理员，每天要处理几十单线下转账确认。用户下单后线下微信转账，上传凭证；管理员需要核对凭证后在系统里"确认付款"，订单才能进入派单。

页面需求：
- 左侧菜单高亮"订单管理 > 待确认付款"
- 主区域是一个可筛选的表格：订单号、下单时间、下单用户、陪玩（若指定）、应付金额 ¥80.00、订单类型（指定/普通）、凭证预览缩略图、操作
- 操作列：【查看凭证】【确认付款】【驳回】
- 点"确认付款"弹抽屉：凭证大图 + 订单详情 + 实际到账金额输入框（默认=应付金额）+ 备注 + 确认/取消按钮
- 键盘快捷键：Y 确认、N 驳回、J/K 上下翻行（需要在页面角落有快捷键提示）
- 表格密度偏紧凑，一屏至少 15 行

请同时给出：默认态、空状态（"今日已全部处理完毕 🎉"）、加载骨架屏、确认弹窗态。中文简体，金额两位小数。
```

### Prompt B：小程序下单页

```
请设计一个微信小程序的"下单"页。技术栈 Taro + NutUI-React-Taro 默认主题。基线 375×812（iPhone X）。

业务背景：用户从陪玩详情页点"下单"进入本页。下单后**不立即支付**（平台不接支付），只生成订单号并显示线下转账说明。

页面需求：
- 顶部：陪玩头像 + 昵称 + 服务项标签
- 表单：
  - 服务类型（从陪玩可选列表下拉）
  - 时长（步进器，0.5h / 1h / 2h / 3h / 4h，最多 10h）
  - 备注（可选，多行）
  - 联系方式（预填微信昵称，可改）
- 金额预览卡：单价 ¥40.00 × 2h = 合计 ¥80.00
- 底部吸底按钮："确认下单"，点击后跳转订单详情页
- 提交后的订单详情页单独一屏：订单号、应付金额、俱乐部收款二维码（占位）、"我已转账，上传凭证"按钮（上传≤5MB jpg/png/pdf）
- 凭证上传后状态提示："已提交，等待管理员确认（通常 5 分钟内）"

风格：NutUI 默认京东风，金额用 ¥80.00 两位小数，文案轻快带点口语。
```

### Prompt C：陪玩订单池 + 抢单

```
请设计微信小程序"陪玩订单池"页。技术栈 Taro + NutUI。

业务背景：陪玩打开 App 后看订单池，是一张张可抢的普通单卡片（同俱乐部内）。抢单是秒杀场景，谁快谁得。

页面需求：
- 顶部：当前工作状态（在线/离线/忙碌/休息）切换器 + 今日接单 3 单 / 今日收入 ¥240.00
- 订单列表（卡片流）：
  - 每张卡：服务类型 + 时长 + 单价 + 合计金额 + 下单时间（几分钟前）+ 用户昵称（打码）+ 备注摘要
  - 右下角【抢单】按钮，按钮带 1 秒进度圈防误触
- 下拉刷新 + 上拉加载
- 空状态："暂时没有可抢订单，在线等等吧"
- 抢单成功后弹窗："抢单成功！请尽快联系用户开始服务"，一键跳转订单详情

风格：NutUI 默认，金额两位小数。
```

> 后续页面（报备、提现、打赏等）按同样模板写 Prompt：**明确技术栈 + 业务背景 + 字段列表 + 核心动作 + 必须态（空/加载/错误）+ 文案风格**。

## 11. 设计不做什么（反向约束）

- ❌ 不设计支付收银台、不要"立即支付"按钮
- ❌ 不自定义品牌色、不加渐变 / 磨砂 / 3D 插画
- ❌ 不设计直播 / 语音房 / 社交动态（MVP 不含）
- ❌ 不做 Dark Mode（MVP 不要）
- ❌ 不做 PC 小程序版本（MVP 只做微信手机端）
- ❌ 不设计陪玩排行榜 / 打赏榜（MVP 不含）
