# 订单规则

## 1. 订单类型

- 指定单：用户选择具体陪玩，下单后定向派发
- 普通单：用户未指定陪玩，进入订单池等待抢单或派单
- 急单：可后续扩展，MVP 不作为核心必做
- 新手单：可后续扩展，MVP 先不做专项机制

## 2. 订单状态

建议使用以下状态：
- `pending_payment`：待付款确认
- `paid_pending_dispatch`：已确认付款，待派发
- `pending_accept`：已派单，待陪玩响应
- `accepted`：陪玩已接单
- `in_service`：服务中
- `pending_report`：待提交报备
- `pending_report_audit`：报备待审核
- `completed`：已完成
- `cancelled`：已取消
- `closed`：异常关闭

## 3. 下单规则

### 指定单
- 用户选择陪玩和服务项创建订单
- 管理员确认到账后，订单进入定向派发
- 指定陪玩应在限定时间内响应
- 超时未响应可转为普通单或管理员人工改派

### 普通单
- 用户创建订单后等待管理员确认付款
- 确认付款后进入订单池
- 符合条件的陪玩可抢单
- 无人接单时管理员可人工派单

## 4. 接单规则

- 陪玩只能接自己所属俱乐部的订单
- 陪玩处于离线、休息或服务中状态时不可接单
- 抢单成功后其他陪玩不可再次接单
- 指定单拒绝不计入普通拒单限制
- 普通派单可设置响应时限，超时后自动失效

## 5. 服务与报备规则

- 陪玩接单后可开始服务
- 服务结束后必须提交报备
- 报备至少包含开始时间、结束时间、时长、服务内容
- 管理员审核通过后订单才可完成
- 报备驳回后允许重新提交

## 6. 取消规则

- 待付款状态可由用户主动取消
- 已确认付款后的取消需管理员介入处理
- 已接单或服务中的订单不允许用户直接取消
- 所有取消操作必须记录取消人和取消原因

## 7. 派单记录

系统应单独记录订单派发过程，包括：
- 派发模式
- 派发对象
- 派发时间
- 响应结果
- 超时或拒绝原因
